So the guard existed, but too late — !tx.result.details runs unconditionally on any tx.result that wasn't caught by the -5 branch, and null.details throws a TypeError, crashing the whole payment-processor fork.

  Answering your specific question — tx.result is null/undefined whenever gettransaction errors with any code other than -5 (which is Bitcoin Core's "invalid or non-wallet transaction id" code, the only one the prior branch checked for). That includes: -8 (invalid txid hex format), -4 (generic wallet error), -1 (misc error), an RPC timeout, or any daemon-specific error code. Any of those leaves tx = {error: {...}, result: null}, and the old code would hit
